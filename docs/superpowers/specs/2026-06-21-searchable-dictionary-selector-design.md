# 学习计划可搜索词书选择器设计

## 背景

管理端“学习计划”页面当前使用原生下拉框选择词书。词书数量增加后，用户只能逐项浏览，难以快速定位目标词书，并且选项中缺少词数信息。

## 目标

- 将原生词书下拉框替换为可搜索的单选选择器。
- 支持按词书名称进行不区分大小写的部分匹配。
- 在候选项和选中状态中同时展示词书名称与词数。
- 保持现有创建学习计划接口和 `dictionaryId` 提交流程不变。

## 交互设计

1. 默认状态显示“选择词书”；已选择时显示词书名称和总词数。
2. 点击选择器后展开浮层，并自动聚焦搜索框。
3. 输入关键词后立即在前端过滤当前已加载的词书列表，不发起额外请求。
4. 每个候选项展示词书名称及“X 词”；点击候选项后完成单选并收起浮层。
5. 搜索无结果时显示“没有匹配的词书”，并允许清空搜索条件。
6. 再次打开选择器时保留当前选中项，但搜索词重置为空，以便重新浏览全部词书。

## 组件边界

新增页面专用的 `SearchableDictionarySelect` 组件，输入为词书列表、当前词书 ID 和变更回调。组件负责展开状态、搜索关键词、候选项过滤和展示；`StudyPlansPage` 继续负责表单状态与创建请求。

组件不访问 API，不维护业务提交状态，因此可独立测试，也不会改变后端数据流。

## 数据流

1. `StudyPlansPage` 继续通过 `api.listDictionaries()` 一次性加载可选词书。
2. 页面把词书列表和 `form.dictionaryId` 传给选择器。
3. 用户选择词书后，组件通过回调返回词书 ID。
4. 页面更新 `form.dictionaryId`，创建计划时仍转换为数字并提交。

## 边界情况与可访问性

- 空词书列表显示“暂无可选词书”，选择器不可执行选择。
- 搜索关键词只包含空白时按未搜索处理。
- 词数缺失时显示 `0 词`，与现有词书页面保持一致。
- 触发器和候选项使用明确的按钮语义，并提供展开状态与可识别名称。
- 支持按 `Escape` 关闭浮层；关闭后焦点返回触发器。

## 测试与验收

- 输入部分词书名后，仅显示名称匹配的候选项。
- 匹配不区分大小写。
- 无匹配结果时显示空状态，清空后恢复全部候选项。
- 选择候选项后，触发器显示名称和词数，页面表单保存对应 ID。
- 创建计划时 API 请求中的 `dictionaryId` 与选中项一致。
- 修改完成后运行管理端测试与生产构建，并按仓库要求重建、重启 `admin-frontend` 容器。
